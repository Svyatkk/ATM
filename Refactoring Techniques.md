# Refactoring Techniques

У ході розробки проєкту було застосовано наступні техніки рефакторингу:

### 1. Extract Service (Виділення сервісу)
Логіка взаємодії з базою даних була винесена з контролерів у окремі сервіси.  
**Приклад:** [`server/src/services/service.user.ts`](./server/src/services/service.user.ts) містить логіку роботи з користувачем, яку викликає `user.controller.ts`. Це зменшує дублювання та спрощує тестування.

### 2. Extract Interface (Виділення інтерфейсу)
Для забезпечення типізації та уникнення "магічних об'єктів" було створено окремі файли інтерфейсів для кожної сутності.  
**Приклад:** папка [`client/src/types/`](./client/src/types/), де кожна сутність (`IUser`, `IBooking` тощо) має свій файл.

### 3. Replace Nested Conditional with Guard Clauses (Заміна вкладених умов на охоронні конструкції)
Замість складних вкладених `if-else`, використовуються швидкі перевірки з негайним викиданням помилки або поверненням результату.  
**Приклад:** У сервісах (`api/user.service.ts`) спочатку перевіряється `!response.ok`, що робить основний код лінійним та читабельним.

### 4. Move Method (Переміщення методу)
Методи обробки даних було переміщено ближче до самих даних або у відповідні доменні модулі.  
**Приклад:** Конфігурація шляхів та заголовків HTTP переміщена у [`client/src/api/config.ts`](./client/src/api/config.ts).

### 5. Rename Variable/Function (Перейменування для ясності)
Назви функцій та змінних було змінено на більш описові, що відображають їхню суть.  
**Приклад:** Використання префікса `I` для інтерфейсів (`IUser`, `IHost`) та чітких дієслів для функцій (`registerUser`, `loginUser`, `fetchOptions`).

### 6. Replace Magic Literal with Symbolic Constant (Заміна магічного літералу символьною константою)
Для уникнення помилок при написанні URL-адрес та статусів, вони були винесені у константи.  
**Приклад:** У файлі [`client/src/api/config.ts`](./client/src/api/config.ts) створено об'єкт `PAGES_URL`, що дозволяє централізовано керувати шляхами сторінок.

### 7. Decompose Conditional (Декомпозиція умовної конструкції)
Складні логічні перевірки розбиваються на окремі змінні з описовими назвами, що полегшує розуміння коду.  
**Приклад:** У [`server/src/services/serivce.house.ts`](./server/src/services/serivce.house.ts) перевірка валідності дат винесена у булеву змінну `hasDates`, що робить логіку запиту до бази даних лінійною.
