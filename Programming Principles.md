## Programming Principles

### 1. Single Responsibility Principle (SRP)
Кожен модуль та компонент відповідає лише за одну чітко визначену задачу.  
- `userService` → HTTP-запити пов'язані з користувачем  
- `bookingService` → логіка бронювань  
- `houseService` → робота з готелями  
- `PageHotel` → відображення готелю та бронювання  
- `DateDropDown` → лише UI для вибору дат  
- `PanelButtons` → лише навігаційні кнопки профілю  

Файли: [`src/api/user.service.ts`](./client/src/api/user.service.ts), [`src/api/booking.service.ts`](./client/src/api/booking.service.ts), [`src/components/DateDropDown`](./client/src/components/DateDropDown)

### 2. Open-Closed Principle (OCP)
Програмні сутності мають бути відкритими для розширення, але закритими для змін.  
- **Routing**: Додавання нових маршрутів у `server/src/index.ts` або нових сторінок у `client/src/app` відбувається шляхом розширення конфігурації/файлової структури без необхідності змінювати основну логіку обробки запитів.
- **Styles**: Використання глобальних CSS-змінних дозволяє змінювати тему оформлення без редагування кожного окремого компонента.

---

### 3. Liskov Substitution Principle (LSP)
Об'єкти в програмі мають бути замінюваними на екземпляри їхніх підтипів без зміни правильності роботи програми.  
- **TypeScript Interfaces**: Будь-який об'єкт, що відповідає інтерфейсу `IUser` (наприклад, дані з БД або мок-дані для тестів), може бути переданий у компоненти профілю без порушення їхньої роботи.

---

### 4. Interface Segregation Principle (ISP)
Типи розбиті на окремі інтерфейси у папці `src/types/` — кожен описує виключно свою сутність без зайвих полів:  
`IUser`, `IHost`, `IRoomType`, `Iroom`, `IBooking`, `IFormRoom`, `ICountry`, `ICity`, `IImage`.  
Жоден інтерфейс не змушує компонент залежати від даних, які йому не потрібні.

Файл: [`src/types/`](./client/src/types/)

---

### 5. Dependency Inversion Principle (DIP)
Модулі високого рівня не повинні залежати від модулів низького рівня. Обидва мають залежати від абстракцій.  
- **Prisma ORM**: Сервіси (наприклад, `auth.service.ts`) залежать від абстракції Prisma Client, а не від конкретної бази даних (PostgreSQL) чи сирих SQL-запитів. Це дозволяє легко змінити БД у майбутньому.
- **API Services**: Компоненти залежать від функцій-сервісів, а не від прямого виклику `fetch`, що дозволяє змінювати логіку запитів в одному місці.
